# Referência: Projetos e prática (sair do tutorial hell)

O "tutorial hell" é o estado de assistir curso atrás de curso, sentir
que aprende, e não conseguir construir nada sozinho. A cura não é mais
tutorial — é construir com o tutorial fechado. Aqui está o protocolo.

## Princípios
- Aprendizado real acontece na dificuldade produtiva: travar, buscar,
  resolver. Tutorial remove exatamente a parte que ensina.
- Terminar um projeto pequeno > começar dez grandes. Escopo de
  iniciante que "termina" ensina mais que ambição que morre na metade.
- Projeto que a pessoa QUER ver pronto sustenta a motivação melhor que
  o projeto "certo" que não interessa a ela.

## Sinais de que a pessoa está no tutorial hell
- Consegue seguir vídeo, não consegue começar com a tela em branco.
- "Já sei o básico" faz meses, mas nunca construiu nada próprio.
- Recomeça cursos de introdução em vez de avançar.
- Tem medo de errar sozinha sem alguém mostrando o próximo passo.

## Método prático para destravar
1. Escolher UM projeto pequeno e pessoal (lista de tarefas, conversor,
   um joguinho de adivinhação, um script que organiza arquivos dela).
2. Escrever em português a lista de coisas que o projeto precisa fazer
   — o escopo. Cortar tudo que não for essencial para a primeira
   versão.
3. Construir a menor versão que funciona (uma funcionalidade só) e
   rodá-la. Depois crescer uma peça por vez.
4. Regra do tutorial: pode consultar para uma dúvida pontual, não para
   copiar o projeto inteiro. Se copiou um bloco, feche a fonte e
   reescreva de memória — se não conseguir, não aprendeu ainda.
5. Quando travar, ir para references/debugar_e_destravar.md antes de
   fugir para outro curso.

## Construir o hábito
- Sessões curtas e frequentes batem maratonas raras. Constância vence.
- Versionar com git desde cedo, mesmo sozinho: `init`, `add`, `commit`
  com mensagem, e commitar a cada pedaço que funciona. Serve de
  histórico e de rede de segurança para experimentar sem medo.
- Guardar os projetos num lugar só (repositório) — ver o próprio
  progresso acumulado combate a sensação de "não saio do lugar".

## Erros comuns
- Escolher primeiro projeto grande demais (uma rede social) e afogar.
- Perfeccionismo: reescrever o começo em vez de terminar a primeira
  versão feia que funciona. Feito > perfeito na fase de aprender.
- Nunca mostrar o código a ninguém por vergonha; feedback acelera.
- Trocar de linguagem/framework ao primeiro tédio, reiniciando o ciclo.

## Limites e encaminhamento
- Se o projeto exige decisões de estrutura de código de sistema maior
  (como organizar módulos, banco, camadas), isso é arquitetura ->
  references/limites.md (Arquiteto de Software).
- Não entregue o projeto pronto para a pessoa "ter um portfólio";
  portfólio copiado não ensina e não sustenta entrevista.
